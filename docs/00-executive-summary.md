# Executive Summary

Metopus is an active, commercially sensitive artist and fan platform. The production source remains private, but this case study documents the public-safe parts of the work: product framing, architecture, cross-platform implementation strategy, testing approach, AI-assisted workflow and publication boundaries.

## What Metopus Is

Metopus is built around direct artist/fan connection. The product combines public artist discovery, fan capture, authenticated app surfaces, native mobile experiences, interactive visual systems and shared product logic across platforms.

The public explanation is deliberately simple:

> A connected home for music communities, from public artist discovery to deeper fan engagement, artist tools and shared product experiences.

## What I Built And Directed

My role covers product direction, system planning, interface direction, implementation support, testing and documentation.

Public-safe work discussed in this repository includes:

- public web and artist-profile strategy
- authenticated web app direction
- native Android and iOS app planning
- shared Rust and WebAssembly architecture
- Supabase/PostgreSQL and Cloudflare-oriented infrastructure decisions
- visual-system and card-surface planning
- safety, onboarding and cross-device testing approach
- AI-assisted development workflow and review process

## Why The Source Is Private

The private repositories include active product code, proprietary mechanics, security-sensitive backend details, generated bindings, app/client code, unpublished assets and unfinished commercial work.

Publishing a source dump would weaken the product and create unnecessary privacy, security and IP risk. This repository therefore acts as a public-safe evidence layer rather than an open-source release.

## What This Case Study Proves

This work is useful evidence for roles that value product thinking, implementation judgement, technical communication, front-end judgement, QA, digital learning, support, creative technology or cross-functional delivery.

It shows:

- how a broad product idea was shaped into a multi-surface architecture
- how public web, app surfaces, native clients and shared Rust logic fit together
- how commercial confidentiality can be handled without hiding the whole story
- how testing has to account for browser, native and device-specific behaviour
- how AI-assisted development can be used responsibly inside a reviewed workflow

## Current Public Status

This is a documentation-first case study. Screenshots, short clips and live walkthrough material are intentionally withheld until a privacy and IP review is complete.

The next public-safe layer is a small set of reviewed visual assets: clean screenshots, redacted app surfaces, public architecture diagrams and optional controlled capture clips that do not reveal private data or protected mechanics.
